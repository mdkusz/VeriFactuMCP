# VeriFactu System Overview

## Purpose

VERI*FACTU is a system defined by the Spanish Tax Agency (AEAT) to allow invoicing software to submit invoice records electronically in real time.

The system does not transmit invoices themselves but **invoicing records generated by the billing system**.

These records allow the tax authority to verify the integrity and traceability of invoices generated by software systems.

The system is defined by Royal Decree 1007/2023 and technical specifications published by AEAT.

## Communication Model

Communication between billing systems and the AEAT is performed using web services.

Characteristics:

Transport protocol:
HTTPS

Message protocol:
SOAP 1.1

Payload format:
XML

Encoding:
UTF-8

Authentication:
Qualified electronic certificate

The message format is fully defined by XML Schema (XSD).

## Message Exchange

All communications follow the same pattern.

1. Client sends an XML message inside a SOAP envelope.
2. AEAT validates the message.
3. AEAT returns a synchronous response.

Possible responses:

1. SOAP Fault  
   The XML structure is invalid.

2. XML Response  
   The message was accepted for processing and the records are validated.

## Types of Operations

Three logical operations exist.

### Registration of invoicing records

Registers a new invoice record.

XML node:
RegistroAlta

Used for:

- normal invoice registration
- correction of a previously sent invoice
- re-submission after rejection

### Cancellation of invoicing records

Cancels a previously submitted record.

XML node:
RegistroAnulacion

### Consultation of submitted records

Retrieves previously submitted invoicing records.

XML node:
ConsultaFactuSistemaFacturacion

This service is only available for voluntary VERI*FACTU systems.

## Structure of a Submission

A submission contains:

Header
List of invoicing records

General structure:

SOAP Envelope
  Body
    Request
      Cabecera
      RegistrosFacturacion

Each submission can contain both:

RegistroAlta
RegistroAnulacion

## Submission Limits

Maximum records per submission:

1000

If more records must be sent, multiple submissions must be performed.

## Invoice Record Identity

Each invoice record is uniquely identified by three fields:

NIF
NumSerieFactura
FechaExpedicionFactura

Combined identifier:

NIF + NumSerieFactura + FechaExpedicionFactura

This identifier is used for:

- duplicate detection
- corrections
- cancellations
- queries

## Result States

Two levels of result exist.

### Global submission result

Correct  
All records accepted.

PartiallyCorrect  
Some records accepted and some rejected, or accepted with errors.

Incorrect  
All records rejected.

### Individual record result

Correct  
Record accepted.

AcceptedWithErrors  
Record accepted but contains admissible errors.

Incorrect  
Record rejected.

## Error Types

Two error categories exist.

Non-admissible errors  
Cause rejection of the record.

Examples:

- schema validation errors
- invalid identifiers
- invalid dates
- invalid business rules

Admissible errors  
The record is accepted but the system reports a warning.

These errors do not prevent the registration of the invoicing record.

## SOAP Fault

If the XML structure is invalid, AEAT returns a SOAP Fault instead of a normal response.

Example structure:

Envelope
  Body
    Fault
      faultcode
      faultstring
      detail

When a SOAP Fault is returned the entire submission must be corrected and resent.

## Pagination in Queries

Queries may return a maximum of 10,000 records.

If more records exist, pagination must be used.

Pagination is performed using the identifier of the last returned record.

## XML Schemas

The message structures are defined by XML schemas published by AEAT.

Main schemas:

SuministroInformacion.xsd
SuministroLR.xsd
ConsultaLR.xsd
RespuestaSuministro.xsd
RespuestaConsultaLR.xsd
RespuestaValRegistNoVeriFactu.xsd
EventosSIF.xsd

These schemas define:

- message structure
- data types
- validation rules
